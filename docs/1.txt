Data Sources Summary

NVD (National Vulnerability Database)
What it is: The US government's official CVE registry. The authoritative source for vulnerability identification and severity scoring.
What we collect:

CVE ID, status, published/modified dates
CVSS score and severity (CRITICAL/HIGH/MEDIUM/LOW)
Description, CWE weakness type

Why we need it: Source of truth for "does this vulnerability exist and how bad is it?"

OSV (Open Source Vulnerabilities)
What it is: Google-maintained database focused on open source packages. Maps vulnerabilities to specific package versions.
What we collect:

CVE/OSV ID linkage
Package name and ecosystem (PyPI, npm, Maven, etc.)
Affected version ranges (introduced in X, fixed in Y)

Why we need it: NVD tells us what is vulnerable. OSV tells us which exact package versions are affected.

Red Hat Security API
What it is: Red Hat's database of how CVEs affect their products and what patches they've released.
What we collect:

Advisory IDs (RHSA-xxxx)
Affected RHEL products and versions
Fix state and patched package versions

Why we need it: If your company runs RHEL servers, this tells you which Red Hat patches to apply.

CISA KEV (Known Exploited Vulnerabilities)
What it is: US government's list of vulnerabilities actively being exploited in the wild.
What we collect:

CVE ID
Date added to list
Ransomware campaign usage flag

Why we need it: Prioritization. A CVE on this list is not theoreticalâ€”attackers are using it right now.
